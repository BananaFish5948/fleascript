# Project Rules for Antigravity Agents — FleaScript

このワークスペース（FleaScript）で活動する全てのエージェントは、以下の絶対防衛ルールを遵守すること。

---

## 0. プロダクト概要（コンテキスト高速復元用）

- **プロダクト名**: FleaScript（フリマ商品説明ジェネレーター）
- **ターゲット**: ネット初心者〜中級者
- **コア機能**: テキストメモ入力 → AI（GPT-4o-mini）による商品説明文の自動生成
- **コスト制約**: 月1万円以内（画像OCRはMVPフェーズ除外）
- **技術スタック**:
  | レイヤー | 技術 |
  |---|---|
  | フロントエンド | Next.js (App Router, TypeScript, Tailwind CSS v4) |
  | バックエンド | Next.js API Routes（同一リポジトリ） |
  | AI | OpenAI `gpt-4o-mini` |
  | DB / ログ | Supabase |
  | デプロイ | Vercel |
  | レートリミット | Supabase `ip_rate_limits` テーブル（自前実装）|

## 1. エンコーディングとファイル操作の死守

- 読み書きする全てのコードファイルは **必ず `UTF-8 (BOMなし)`** として扱うこと。
- コマンドラインからの出力リダイレクト（`>`, `>>`, PowerShellの `Out-File` 等）は原則禁止。
- ファイルの作成・編集は必ず `write_to_file`, `replace_file_content`, `multi_replace_file_content` を使用すること。

## 2. 大規模置換の禁止と安全な差分修正

- 一度に数十行・数百行を超える強引なコード置換は禁止。
- 局所的な修正は `replace_file_content` / `multi_replace_file_content` を使い、ファイル全体を再生成しないこと。

- **【Vercelブロック回避の絶対ルール】**: コミット時は必ず `--author="BananaFish5948 <k.shin.0103@gmail.com>"` を付与すること。省略は厳禁（Vercel無料枠制限に抵触する）。
- **【コミットフォーマット】**:

  - **本番環境（Vercel）へのデプロイトリガーとなる `master` ブランチへのプッシュおよびマージは、マスターから明示的なデプロイ許可（Goサイン）が出た場合のみ実行すること。** マスターの事前承認なしに、勝手に `master` へプッシュして自動リリースを行うことは厳禁とする。

## 4. ドキュメント更新の限定化（トークン節約）

- 単なる進捗ログのドキュメント化は禁止。ソースコード自体を最新ステートとして扱う。
- `docs/handover_context.md` の更新は以下の条件のみ:
  1. 次のAIが確実にハマる重大な罠（Gotchas）を発見した時
  2. マスターから「引き継ぎ書を作れ」と明示的に指示された時

## 5. AI共通開発ルール

- **【自己修復のブレーキ】**: エラー自動修復ループは **最大3回まで**。超えた場合はマスターに報告。
- **【疎結合の維持】**: API RoutesとUI Componentsの責務を明確に分離すること。
- **【コスト意識の徹底】**: OpenAI APIコールは必ずトークン数・モデルを意識し、不必要な呼び出しを排除すること。

## 6. チケット駆動開発運用プロトコル（Five Agents System）

当プロジェクトは、四姉妹（コーディネーター・フロントエンド・バックエンド・QA等）に技術監査役（Auditor）を加えた5つのエージェントによって自律的に開発を推進する。

| エージェント | 役割 | 主な担当作業 |
|---|---|---|
| **Coordinator** | 管制塔（ジェミリア） | 実装計画・タスクチケット生成・進行管理・最終レビュー |
| **Worker** | 実行部隊（エンジニア等） | コーディング・テスト・差分作成・チケット完了報告 |
| **Auditor** | 技術監査役（クロード） | 複雑なバグの死角探知・解決策の提示（直接修正なし） |

### チケット保管場所
- `workspaces/worker_frontend/` — フロントエンド系タスク
- `workspaces/worker_backend/` — バックエンド系タスク
- `workspaces/TICKET_TEMPLATE.md` — テンプレート（コピー使用、上書き禁止）

## 10. 実装と品質保証の厳格化ルール（Gemini用うっかりミス防止策）

**全てのコーディング作業において、以下の思考・計画フェーズを義務付ける。**
ロングコンテキストモデル特有の「注意力の拡散」や「余計な最適化」を防ぐため、以下のステップとルールを「必ず」厳守すること。

### 10-1. 思考・計画フェーズ（コード生成前の義務）
タスクを受け取ったら、いきなりコードを書いてはならない。まずは以下の構成で「実装計画（Implementation Plan）」を提示し、**マスターまたは監査役（Auditor）の承認（Goサイン）を待つこと**。
1. **影響範囲の特定**: 修正が必要なファイルと、それに依存する（型定義やインポートで影響を受ける）ファイルをすべてリストアップせよ。
2. **破壊的変更の有無**: 既存のインターフェース、スキーマ、共通関数のシグネチャを変更するかどうかを明記せよ。
3. **うっかりミス警戒ポイント**: このタスクにおいて、あなたが「見落としがちなインポート、型定義の矛盾、エッジケース」を3つ事前予想して書き出せ。

### 10-2. コード生成フェーズの厳格なルール
承認を得た後、コードを生成する際は以下を遵守する。
- **型定義の完全性**: 型の不一致や、雑な `any` （またはそれに準ずる曖昧な型）の使用は一切禁止。
- **インポートの完全性**: 新しい関数や型を使用する場合、インポート文（import/require）が正しく追加・修正されているか、1ファイルずつセルフチェックすること。
- **コードの省略禁止**: 「// 既存のコード...」といった省略は原則禁止。コンテキストウィンドウの余裕を活かし、修正対象のブロック全体、またはファイル全体を完全に一貫した状態で出力すること（トークン節約を優先してコードを破壊しない）。
- **変更境界の厳守**: 指示されていないファイルや型定義は「絶対に変更しない」。ついでに最適化しようとする親切心はバグの元となるため厳禁。

### 10-3. 自己検証と修正時のアプローチ
- コードを確定させる直前に、「呼び出し元の引数は一致しているか」「非同期処理のハンドリングは漏れていないか」を心の中でトリガーし、問題がない場合のみ実装完了を宣言せよ。
- マスターからミスを指摘された場合は、すぐにコードを再生成するのではなく、**「なぜ間違えたか」を1行で説明（Self-Correction）してから**修正版を出力すること。

## 11. マスター向け運用Tips（マスターチップス）

### 11-1. コンテキスト分断（ブラインド監査）の手順
Antigravity IDEの仕様上、同じチャットセッション内で会話を続けると、QA担当エージェントが「実装担当の意図や言い訳」を先読み（忖度）してしまい、レビューが甘くなる「同調バイアス」が発生します。これを防ぎ、厳格なコードレビューを実現するためには、以下の手順で「ブラインド監査」を実施してください。

**【手順】**
1. **コード（事実）の抽出**: 現在の作業チャットで、実装担当に `git diff` を出力させるか、修正したコード全体を出力させます。（※もしVercel等でビルドエラーが出ている場合は、そのエラーログをコピーします）
2. **監査用別ウィンドウの立ち上げ（推奨）**: VS Codeの仕様上、同じフォルダを通常の手順で複数開くことはできません。コマンドパレット（`Ctrl + Shift + P`）を開き、**`Workspaces: Duplicate As Workspace in New Window`** （日本語環境: `新しいウィンドウにワークスペースとして複製`）を実行してください。完全に同じプロジェクトが別ウィンドウで開きます。そこでAntigravity IDEの「新しいチャット（New Chat）」を開始します。
3. **初見レビューの強要**: 新しい監査用チャット画面に、手順1で抽出した「コード」や「エラーログ」だけを直接貼り付けます。今までの経緯や言い訳は一切入力しません。
4. **スキル指定のメンション**: メッセージ送信時、必ず文頭で `@qa-tester` または `@auditor` とメンション（スキル指定）し、「このコードのバグや懸念点を、初見の状態で厳しく監査しろ」と指示を投げます。

**💡 なぜこれが効くのか？（メリット）**
過去の文脈（コンテキスト）が物理的に遮断されるため、QA・Auditorは純粋に「提示された文字列（事実）」だけを見て冷徹に審査せざるを得なくなります。馴れ合いによる見落とし（Vercelでのビルド落ち等）を防ぐ最強のサバイバル術です。

**💡 監査役の運用ルール（競合・コンフリクト防止）**
別ウィンドウでプロジェクトを開いて監査する場合、同じファイルを監査役と実装担当が同時に編集すると保存競合が起きます。そのため、**「監査役（QA / Auditor）には絶対にコードを直接編集・出力させず、テキストによる指摘に留め、修正の適用は必ずメインウィンドウの実装担当に行わせる」** というルールを徹底してください。
